# PRD: 폐쇄망 기반 공공기관 민원 응답 지원 sLLM 시스템

## 1. 문서 정보
- 제품명: 민원 응답 지원 AI (On-Prem sLLM + Semantic Search)
- 문서 버전: v1.0
- 작성일: 2026-03-06
- 대상 조직: 지자체 및 공공기관 민원 담당 부서
- 배포 환경: 폐쇄망/온프레미스

## 2. 제품 정의
- 문제: 민원 답변 초안 작성 시 유사 사례 탐색과 문서 작성에 시간이 많이 소요되고, 담당자별 품질 편차가 발생함.
- 해결: 민원 요지를 입력하면 의미 기반 유사 사례를 검색하고, 기관 표준 템플릿을 반영한 답변 초안을 자동 생성하는 시스템 제공.
- 핵심 가치: `보안성`, `일관성`, `신속성`, `비용 효율성`.

## 3. 목표 및 비목표
### 목표
1. 민원 답변 초안 작성 리드타임 단축.
2. 기관 정책/톤앤매너를 반영한 답변 품질 표준화.
3. 외부 클라우드 없는 폐쇄망 운영 보장.
4. CPU/GPU 환경 모두에서 실사용 가능한 추론 성능 확보.

### 비목표
1. 완전 자동 발송(담당자 검토 없는 자동 민원 회신).
2. 비공개/내부 민원 원문의 무제한 학습 자동화.
3. 멀티모달(음성/이미지) 민원 처리.

## 4. 사용자 및 시나리오
- 1차 사용자: 민원 담당 공무원
- 2차 사용자: 관리자(정책 템플릿/권한/로그 관리), 운영자(모델/인덱스 배포 및 성능 관리)

### 핵심 시나리오
1. 담당자가 민원 요지를 입력한다.
2. 시스템이 유사 민원-답변 사례를 상위 N건 검색한다.
3. 시스템 프롬프트 + 검색 결과를 기반으로 답변 초안을 생성한다.
4. 담당자가 문구를 수정/보완 후 복사해 업무 시스템에 반영한다.
5. 수정 이력은 익명화 저장되어 향후 개선 데이터로 활용된다.

## 5. 기능 요구사항 (Functional Requirements)
### FR-01 데이터 수집/정제 파이프라인
- 공개 민원 게시판 대상 수집 크롤러 제공.
- 중복 제거, 포맷 표준화, 비식별화 수행.
- 데이터셋 버전 관리(`dataset_version`) 및 품질 리포트 생성.

수용 기준:
1. 수집 실패율 5% 이하.
2. 비식별화 규칙(이름/연락처/주소/식별번호) 적용 후 샘플 점검 통과율 99% 이상.
3. 동일 데이터 재수집 시 중복 제거 정상 동작.

### FR-02 학습 및 모델 관리
- 오픈소스 sLLM 기반 SFT 파이프라인 제공.
- 모델 버전 관리(`model_version`, 학습 데이터 버전 매핑).
- 오프라인 평가(정확성/정중성/근거 반영) 리포트 생성.

수용 기준:
1. 모델 아티팩트 재현 가능(동일 설정 재학습 가능).
2. 베이스라인 대비 품질 지표 개선 확인.
3. 롤백 가능한 버전 체계 제공.

### FR-03 의미 기반 검색(FAISS)
- 민원 질의 임베딩 생성 및 벡터 인덱싱.
- 유사 사례 Top-K 검색 및 점수 반환.
- 지자체/기관 단위 필터 검색 지원.

수용 기준:
1. Top-K 검색 응답시간 목표 내 충족(성능 항목 참조).
2. 오프라인 평가셋 기준 검색 적합도 기준점 충족.
3. 인덱스 증분 업데이트 가능.

### FR-04 답변 생성 엔진
- 시스템 프롬프트 템플릿(문체, 구조, 금지표현, 안내문구) 적용.
- `요지-근거-안내` 구조의 초안 생성.
- 근거 사례 참조 ID를 함께 제공.

수용 기준:
1. 필수 구조(요지/근거/안내) 누락률 3% 이하.
2. 금지표현 규칙 위반률 1% 이하.
3. 담당자 수정 후 사용 가능 판정 비율 목표 달성(예: 70% 이상).

### FR-05 UI (Streamlit)
- 민원 입력창, 생성 결과창, 수정/복사 기능 제공.
- 유사 사례 참조 패널(선택) 제공.
- 결과 재생성/검색 K값 조정/템플릿 선택 기능 제공.

수용 기준:
1. 비전문가 사용자 기준 30분 내 기본 사용 습득.
2. 주요 액션(입력, 생성, 복사) 3클릭 이내 수행.
3. 오류 발생 시 사용자 친화적 메시지 제공.

### FR-06 운영/관리 기능
- 접근권한 분리(원문 접근 vs 학습 데이터 접근).
- 감사 로그(조회/생성/수정/복사 이벤트) 기록.
- 모델/인덱스/템플릿 버전 상태 조회 대시보드 제공.

수용 기준:
1. 역할별 권한 정책 준수.
2. 감사 로그 보존 정책 충족.
3. 배포 후 헬스체크 가능.

## 6. 비기능 요구사항 (NFR)
### 보안/컴플라이언스
1. 외부 인터넷 미연결 환경에서 전체 기능 동작.
2. 저장/로그 내 개인정보 마스킹.
3. 관련 지침(개인정보 보호, 공공데이터 활용) 준수 체크리스트 운영.

### 성능
1. 검색 응답시간: P95 1.5초 이내.
2. 초안 생성시간: P95 8초 이내(GPU 기준), P95 20초 이내(CPU 기준).
3. 동시 사용자: 최소 20명 기준 성능 저하 허용 범위 내.

### 가용성/운영성
1. 업무시간 기준 가용성 99.5% 이상.
2. 장애 시 모델/인덱스 롤백 절차 30분 내 수행 가능.
3. 오프라인 설치 패키지(Docker)로 재배포 가능.

### 품질
1. 답변 형식 준수율 95% 이상.
2. 정책 근거 반영률 목표치 달성.
3. 사용자 만족도(담당자 설문) 5점 척도 평균 4.0 이상 목표.

## 7. 시스템 범위 및 아키텍처
- 입력: 민원 요지 텍스트
- 처리:
1. 임베딩 생성
2. FAISS Top-K 검색
3. 프롬프트 조합
4. sLLM 생성
5. 후처리(금지표현/형식 검사)
- 출력: 편집 가능한 답변 초안 + 참조 사례
- 구성요소:
1. Data Pipeline (Crawler/ETL/De-ID)
2. Training Pipeline (SFT/Eval/Registry)
3. Inference Service (Ollama/ONNX Runtime/TensorRT)
4. Retrieval Service (FAISS)
5. UI Service (Streamlit)
6. Governance (RBAC/Audit/Masking)

## 8. KPI / 성공 지표
1. 평균 처리시간 30% 이상 단축.
2. 담당자 수정량(문장 단위) 25% 이상 감소.
3. 답변 품질 편차(팀 간) 20% 이상 축소.
4. 개인정보 노출 사고 0건.
5. 월간 활성 사용자 비율 80% 이상.

## 9. 릴리스 계획
### Phase 1 (PoC, 6~8주)
1. 단일 기관 데이터셋 구축.
2. 검색 + 초안 생성 최소기능(MVP) 구현.
3. 폐쇄망 단독 배포 검증.

### Phase 2 (Pilot, 8~10주)
1. 템플릿 고도화 및 품질평가 체계 도입.
2. ONNX/TensorRT/양자화 적용.
3. 2~3개 부서 사용자 테스트.

### Phase 3 (Production, 6~8주)
1. 권한/감사/운영 대시보드 정식화.
2. 기관별 정책 룰셋 분리 운영.
3. 배포 표준화(Docker 패키지, 운영 매뉴얼).

## 10. 리스크 및 대응
1. 데이터 품질 편차.
- 대응: 정제 규칙 템플릿화, 샘플링 검수 프로세스 운영.
2. 폐쇄망 GPU 자원 부족.
- 대응: INT4/INT8 양자화, CPU fallback 모델 준비.
3. 정책 변경 빈번.
- 대응: 프롬프트/룰셋 분리형 설계로 즉시 반영.
4. 사용자 신뢰 부족.
- 대응: 참조 사례와 생성 근거를 함께 노출.

## 11. 운영 정책
1. Human-in-the-loop 의무화(최종 답변은 담당자 승인 후 사용).
2. 모델 업데이트 시 사전 검증셋 통과 필수.
3. 로그/데이터 보존기간 및 파기정책 준수.
4. 월간 품질 리뷰(오탐/부적절 표현/정책 미반영 사례) 시행.

1. 초기 대상 기관 수 및 데이터 사용 범위 확정 필요.
2. 폐쇄망 하드웨어 스펙 최소 기준 확정 필요.
3. 공공기관 내부 결재/보안성 심사 절차와 일정 확정 필요.
4. 민원 유형별 템플릿 우선순위 확정 필요.
