# OSS 주요 라이선스 비교 분석

## 개요

오픈소스 프로젝트를 시작할 때 가장 중요한 결정 중 하나는 라이선스 선택입니다. 이 문서는 가장 널리 사용되는 OSS 라이선스 4가지를 비교 분석합니다.

#### 특징

```

#### 의무 사항
```
! 저작권 고지와 라이선스 명시 필수
! 수정 사항 명시 필수
! 파생 코드도 GPL 3.0으로 공개 (상호주의)
! 설치 정보 제공 (DRM 우회 방지)

#### 제약 사항
```
✗ 파생 작품도 GPL이어야 함 (강제)
✗ 독점 소프트웨어와 통합 불가
✗ 서브라이선싱 시 GPL 유지
✗ 상업용 패키징 이용 가능
```

#### 상호주의(Copyleft) 설명
```
GPL 코드를 포함한 모든 파생 코드는
반드시 GPL로 공개되어야 합니다.

예시:
GPL 라이브러리 + 우리 코드 = GPL 공개 의무
```

## 라이선스 선택 결정 플로우

```
┌─ 파생 코드가 오픈소스여야 하나?
│  ├─ YES → GPL 3.0
│  └─ NO → 다음 단계
├─ 특허 보호가 필요한가?
│  ├─ YES → Apache 2.0
│  └─ NO → 다음 단계
├─ 법적 명확성이 필요한가?
│  ├─ YES → BSD 3-Clause
│  └─ NO → MIT
```

## 라이선스별 호환성

### 혼합 가능한 경우
```
✓ MIT + MIT = MIT
✓ BSD 3-Clause + BSD 3-Clause = BSD 3-Clause
✓ MIT + BSD 3-Clause = MIT 또는 BSD (선택)
✓ Apache 2.0 + Apache 2.0 = Apache 2.0
```

### 혼합 불가능한 경우
```
✗ GPL 3.0 + MIT ≠ 가능 (GPL이 종속)
✗ GPL 3.0 + Apache 2.0 ≠ 호환 불가
  (Apache 2.0의 특허 조항과 GPL 비호환)
```

## 현재 프로젝트의 라이선스 선택 사유

**AI-OSS 프로젝트는 MIT 라이선스를 선택했습니다.**

### 선택 근거
1. **교육 목적**
   - 학기 중 협업 학습 프로젝트
   - 최대한 자유로운 활용 권장

2. **빠른 시작**
   - 라이선스 이해도 높은 것보다 진행속도 우선
   - 법적 복잡도 최소화

3. **커뮤니티 개방성**
   - 모든 형태의 기여 환영
   - 파생 프로젝트의 자유로운 창작 허용

4. **광범위한 채택**
   - MIT는 가장 널리 알려진 라이선스
   - 학생들이 이미 경험한 라이선스

### 향후 고려사항
```
프로젝트 성숙도 상승 시:
- 특허 보호 필요 → Apache 2.0 전환 검토
- 오픈소스 기여 → GPL로 변경 고려
- 엔터프라이즈 → Apache 2.0 + 상업 라이선스 이중 구조
```

## 라이선스 변경 절차

### 변경 가능 조건
```
1. 모든 기여자 동의 필수
2. 이전 버전은 기존 라이선스 유지
3. 신규 버전부터 새 라이선스 적용
```

### 변경 불가 상황
```
1. 기여자가 많을 경우 동의 어려움
2. 제3자 라이선스 포함된 경우
3. 이미 배포된 버전
```
