# Inner Source 도입 로드맵

## Inner Source란?

**Inner Source (내부 오픈소스)는 조직 내부에서 오픈소스 개발 방식을 적용하는 것입니다.**

```
일반 OSS
┌─────────────────────────────────┐
│ 공개 저장소                      │
│ (누구나 접근 가능)              │
└─────────────────────────────────┘

Inner Source
┌─────────────────────────────────┐
│ 내부 저장소                      │
│ (직원만 접근, OSS 방식 운영)    │
└─────────────────────────────────┘
```

### Inner Source의 핵심 원리

1. **투명성**: 모든 코드와 토론이 공개
2. **개방성**: 누구나 기여 가능
3. **멘토링**: 경험 많은 사람이 신입 가이드
4. **신뢰**: 코드 리뷰 기반 품질 보증

## Inner Source의 효과

### 개발 조직 차원
```
효율성 ↑
- 부서 간 코드 공유로 중복 개발 감소
- 내부 라이브러리 활용으로 개발 속도 증가
- 자동화 도구 공유로 업무 효율화

품질 ↑
- 광범위한 코드 리뷰로 버그 감소
- 다양한 관점의 검증
- 최선의 사례(Best Practice) 공유

지식 축적 ↑
- 문서화 의무로 학습 자산 증가
- 암묵적 지식의 명시화
- 조직 신입의 학습 곡선 단축
```

### 직원 차원
```
기술 성장 ↑
- 다양한 코드 리뷰 경험
- 다른 팀의 기술 학습
- 리더십 기회 (Open Source 방식)

만족도 ↑
- 투명성과 신뢰 기반 문화
- 기여의 시각적 인정
- 자율성과 책임감

경력 개발 ↑
- 오픈소스 경험 포트폴리오
- GitHub 프로필 구축
- 외부 프로젝트 기여 자신감
```

### 비즈니스 차원
```
비용 절감
- 중복 개발 제거
- 외부 라이브러리 대체 내부 솔루션
- 유지보수 효율화

위험 감소
- 코드 공개로 보안 이슈 조기 발견
- 버스 팩터 개선 (한 사람 의존도 감소)
- 지식 손실 방지

경쟁력 강화
- 우수 인재 채용 용이
- 기술력 입증 (투명성)
- 생태계 영향력 확대
```

## Inner Source 도입 로드맵

### Phase 1: 준비 (1-2개월)

#### 1.1 경영진 이해와 지지 확보

**액션 아이템**
- [ ] Inner Source 개념과 효과 설명 자료 작성
- [ ] 경쟁사/선례 사례 분석 (Microsoft, Uber 등)
- [ ] ROI 계산 및 제시 (예: 중복 개발 20% 감소 시 연간 XX억 절감)
- [ ] 경영진 워크숍 개최

**체크포인트**
```
Q: 경영진이 Inner Source의 가치를 이해하는가?
Q: 예산과 인원 배정이 가능한가?
Q: 시범 부서 선정이 되었는가?
```

#### 1.2 기업 문화 평가

**평가 항목**
- [ ] 현재 협업 문화 수준 (Siloed vs. Collaborative)
- [ ] 코드 리뷰 문화 존재 여부
- [ ] 문서화 수준
- [ ] 변화에 대한 개방성
- [ ] 신뢰 기반 인간관계

**결과 해석**
```
점수 > 70점: 도입 용이
점수 50-70점: 문화 개선 필요
점수 < 50점: 충분한 준비 기간 필요
```

#### 1.3 거버넌스 구성

**조직 구조 설계**
```
Inner Source 위원회 (경영)
├─ Chief Open Officer
├─ 각 부서 대표
└─ HR, 법무 대표

Inner Source 팀 (실행)
├─ Inner Source 관리자
├─ 개발자 커뮤니티 리더
├─ 문서 담당
└─ 도구 담당
```

**역할과 책임**
| 역할 | 책임 |
|------|------|
| Chief Open Officer | 전사 Inner Source 전략 |
| 관리자 | 프로젝트 우선순위, 정책 |
| 코드 리뷰어 | 기술 수준 유지 |
| 멘토 | 신참 기여자 가이드 |

#### 1.4 정책 수립

**정책 항목**
- [ ] Inner Source 정의 및 가이드라인
- [ ] 지적 재산권 정책 (기여자 권리 보호)
- [ ] 라이선스 정책 (내부/외부 배포)
- [ ] 보안 정책 (민감 정보 관리)
- [ ] 일과 시간 정책 (Inner Source 기여 시간)

**샘플 정책**
```
1. Inner Source 기여는 정규 업무로 인정
2. 월 근무 시간 중 10-15%를 Inner Source에 할당
3. 모든 코드는 라이선스 명시 필수
4. 보안 정보는 별도 저장소 관리
5. 기여자는 CONTRIBUTORS 파일에 인정
```

### Phase 2: 시범 운영 (3-4개월)

#### 2.1 시범 프로젝트 선정

**선정 기준**
```
✓ 다중 팀이 사용하는 프로젝트
✓ 업무 요청이 빈번한 영역
✓ 기술 부채가 있는 부분
✓ 리더십이 강한 팀

예시 프로젝트:
- 공통 개발 라이브러리
- 빌드 자동화 도구
- 모니터링 플랫폼
- 테스트 프레임워크
```

#### 2.2 시범 팀 구성

**팀 구성**
```
시범 프로젝트 리더 1명 (기업 문화 개방적)
핵심 개발자 3-5명 (다양한 팀 소속)
이해관계자 2-3명 (사용 부서 대표)
```

**교육 프로그램**
- [ ] Inner Source 기본 교육 (2시간)
- [ ] 깃 및 GitHub 워크플로우 (3시간)
- [ ] 코드 리뷰 베스트 프랙티스 (2시간)
- [ ] 문서화 표준 (1시간)

#### 2.3 도구 구성

**필수 도구**
```
VCS (Version Control System)
├─ GitHub Enterprise 또는 GitLab
├─ 접근 제어 (브랜치 보호, 리뷰 요구)
└─ 감사 로그 (컴플라이언스)

협업 플랫폼
├─ 토론 (Discussions/Issues)
├─ 프로젝트 관리 (Project Boards)
└─ 위키 (문서화)

CI/CD
├─ 자동 테스트
├─ 코드 품질 검사
└─ 보안 스캔
```

#### 2.4 첫 3개월 목표

**마일스톤**
```
월 1: 팀 구성, 도구 설정, 교육 완료
월 2: 첫 컨트리뷰션 10건, 코드 리뷰 문화 형성
월 3: 컨트리뷰션 누적 30건, 외부 팀 참여 시작
```

**성공 지표**
- [ ] 시범 팀 외 최소 2개 팀의 기여
- [ ] 평균 코드 리뷰 시간 < 24시간
- [ ] 문서 업데이트 비율 > 50%
- [ ] 팀 만족도 > 3.5/5

### Phase 3: 확산 (5-8개월)

#### 3.1 추가 프로젝트 온보딩

**온보딩 프로세스**
```
1. 프로젝트 신청 → 심사 위원회 검토
2. 조건 확인
   - Inner Source 정책 준수 의사
   - 최소 2명 이상의 관리자
   - 문서화 계획 수립
3. 온보딩 교육 제공
4. 메토링 기간 (2-3개월)
5. 공식 Inner Source 프로젝트 인증
```

#### 3.2 커뮤니티 구축

**커뮤니티 활동**
```
월간 Inner Source 밋업
├─ 프로젝트 쇼케이스
├─ 베스트 프랙티스 공유
├─ 기술 워크숍
└─ 네트워킹

분기별 Inner Source Day
├─ 전사 행사
├─ 주요 컨트리뷰터 표창
├─ 키노트 발표
└─ 핸즈온 워크숍

온라인 커뮤니티
├─ Slack/Teams 채널
├─ 위키/드롭박스 공동 문서
├─ 메일링 리스트
└─ 카테고리별 토론 포럼
```

#### 3.3 문화 강화

**문화 체계화**
- [ ] Inner Source 베스트 프랙티스 문서화
- [ ] 프로젝트별 CONTRIBUTING.md 템플릿 제공
- [ ] 코드 리뷰 체크리스트 표준화
- [ ] 신 입사자 온보딩에 Inner Source 포함

**인센티브 프로그램**
```
개인 인센티브
├─ 월간 Top Contributor 인정
├─ 년간 누적 기여도 시상
├─ 리더십 스킬 인증
└─ 경력 기술서에 Inner Source 기여 기록

팀 인센티브
├─ 우수 Inner Source 프로젝트 상
├─ 분기별 팀 크레딧 부여
└─ 개발 도구 우선 지원

조직 인센티브
├─ 급여 평가에 Inner Source 기여도 반영
├─ 승진 기준에 개방성 포함
└─ 팀 간 평가 고려
```

#### 3.4 확산 목표

**6개월 차 마일스톤**
```
Inner Source 프로젝트: 10개 이상
활성 기여자: 50명 이상
월 코드 리뷰: 500건 이상
부서 참여: 70% 이상
```

### Phase 4: 최적화 및 성숙 (9-12개월)

#### 4.1 메트릭 기반 개선

**측정 항목**
```
개발 효율
├─ 중복 코드 감소율
├─ 버그 발견 시간 단축
├─ 기능 구현 속도

코드 품질
├─ 코드 리뷰 커버리지 (%)
├─ 테스트 커버리지 (%)
├─ 보안 이슈 발견 수

조직 효과
├─ 직원 만족도
├─ 이탈율 변화
├─ 신입 온보딩 시간
```

**데이터 수집**
```
자동 수집 (도구)
- GitHub API를 통한 기여도 통계
- CI/CD 파이프라인 성공률
- 코드 커버리지 리포트

수동 조사
- 분기별 만족도 설문
- 연간 조직 건강도 평가
- 인터뷰 및 피드백
```

#### 4.2 거버넌스 고도화

**정책 업데이트**
- [ ] 경험 기반 정책 개선
- [ ] 부서별 맞춤형 가이드라인
- [ ] 보안 및 규정 강화
- [ ] 외부 오픈소스 기여 가이드라인

**리더십 다층화**
```
Level 1: Code Contributor (누구나)
Level 2: Reviewer (경험 2년+, 추천)
Level 3: Maintainer (팀 리더)
Level 4: Inner Source Leader (회사 차원)
```

#### 4.3 외부 오픈소스로의 확대

**외부 기여 프로그램**
```
정책
├─ 회사 시간의 10%는 외부 OSS 기여 가능
├─ 회사 분야와 관련된 프로젝트 우선
└─ 법무 승인 후 진행

지원
├─ 외부 기여자 교육
├─ 컨퍼런스 참가 지원
├─ 기여 시간 인정
└─ 오픈소스 라이선스 교육
```

#### 4.4 성숙 단계 목표

**1년 차 최종 마일스톤**
```
Inner Source 프로젝트: 20개 이상
활성 기여자: 100명+ (전체 개발자 50%)
월 코드 리뷰: 1,000건+
개발 생산성 향상: 15-25%
코드 품질 개선: 버그 30% 감소
직원 만족도: 4.0/5 이상
외부 OSS 기여: 50명 이상 참여
```

## Inner Source 도입 시 주의사항

### 일반적인 실패 요인

```
1. 경영진의 이해 부족
   ├─ 증상: 시간 배정 안 함, 리더십 부족
   ├─ 예방: 지속적인 교육 및 ROI 제시
   └─ 해결: 분기별 경영진 보고

2. 기존 문화와의 충돌
   ├─ 증상: 비공개 개발 계속, 리뷰 거부
   ├─ 예방: 문화 평가 후 단계적 도입
   └─ 해결: 리더의 솔선, 인센티브

3. 도구 부족
   ├─ 증상: 충분한 GitHub Enterprise 라이선스 부족
   ├─ 예방: 사전 예산 확보
   └─ 해결: 오픈소스 도구로 대체

4. 참여 부족
   ├─ 증상: 컨트리뷰션 정체, 낮은 만족도
   ├─ 예방: 초기 성공 사례 축적, 보상
   └─ 해결: 의무 참여 시간 설정

5. 보안/규정 이슈
   ├─ 증상: 민감 정보 유출, 라이선스 위반
   ├─ 예방: 정책 수립, 도구 설정 (접근 제어)
   └─ 해결: 보안 감사, 교육 강화
```

### 성공 조건

```
✓ 경영진의 진정한 지지
✓ 리더의 모범 (관리자의 솔선적 기여)
✓ 명확한 정책과 가이드라인
✓ 적절한 도구 투자
✓ 문화 변화에 대한 시간 할당
✓ 성과 인정 및 보상 시스템
✓ 점진적인 확대 (한 번에 모두 아님)
```
